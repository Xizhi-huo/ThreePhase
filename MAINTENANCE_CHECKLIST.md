## 1. 维护边界总原则

### 1.1 总目标
- 当前阶段 `不新增功能`。
- 当前阶段只做两类事：
  - 提高代码可读性
  - 提高代码可靠性
- 当前阶段重构的终极目的之一，是 `剥离 UI 与业务/物理/评分逻辑`，为未来可能的 `React + Tauri` 或其他 UI 框架替换保留条件。

### 1.2 文件大小标准
| 等级 | 标准 |
|---|---|
| 优秀 | `<= 300` 行 |
| 可接受 | `301 - 500` 行 |
| 需要拆分 | `> 500` 行 |
| 高风险 | `> 800` 行 |

### 1.3 必须遵守的工程边界
- 不再往大文件里继续堆新逻辑。
- 不再新增上帝类。
- 不再新增巨石函数。
- 不再新增 `physics -> ui` 的直接调用。
- 不再新增 `controller -> 具体 UI 控件` 的直接写入。
- 不再新增大范围 `try/except Exception` 静默吞异常。
- 不再新增长期保留的过渡死代码。
- 每次重构必须同步删除旧实现，不能长期双轨并存。
- 所有后端计算结果只能写入 `SimulationState / RenderState / AssessmentResult` 这类状态对象。
- UI 只能读取状态刷新自己，不能反向污染业务状态。
- Controller 只负责命令下发和编排，禁止新增 `ctrl.xxxWidget.setText()/setValue()` 一类直接控件写入。
- 重构核心逻辑前，必须先具备最小黑盒回归验证能力；没有验证保护的重构，不进入核心逻辑。

### 1.4 每轮迭代固定动作
- 至少拆出 `1` 个明确职责。
- 至少让 `1` 个高风险文件明显变小。
- 至少删除 `1` 批旧代码、重复代码或兼容残留。
- 必须同步更新：
  - `README.md`
  - `context.md`
  - 本文件
- 必须记录：
  - 本轮完成项
  - 本轮未完成项
  - 下一轮起点

### 1.5 禁止事项
- 禁止“顺手加功能”。
- 禁止“只抽函数，不删旧逻辑”。
- 禁止“因为赶进度继续把逻辑塞回 app/main.py 或 ui/test_panel.py”。
- 禁止“未记录进度就结束本轮重构”。

## 二、Repomix 与清单脱节问题

你的 repomix 导出中 **`app/main.py` 和 `ui/test_panel.py` 都不存在**（前者只有 `app/__init__.py`，后者在目录树中完全找不到，`ui/main_window.py` 也被标记为 binary 跳过了）。这两个文件恰好是清单中标注的"最大风险文件"。

- 要么这些文件被 `.repomixignore` 排除了（因为太大），但那样 AI 在后续对话中根本看不到它们的代码

## 四、需要进一步细化的内容

**1. 缺少阶段完成的明确 Gate（门禁条件）**

每个阶段有"完成标准"，但标准偏定性。比如 20-40% 的完成标准是「`app/main.py` 不再承载故障、考核、UI 请求、相序解析四类细节」——怎么判定"不再承载"？建议加一个可验证的条件，比如：

> 完成标准验证方式：`grep -c "def.*fault\|def.*assess\|def.*ui_request\|def.*phase.*order" app/main.py` 返回 0

**2. 缺少 `ctrl` 引用传递策略**

通读代码后发现一个贯穿全项目的架构问题：几乎所有 service 的 `__init__` 都接收 `ctrl` 参数并在内部直接访问 `self.ctrl.xxx`。这意味着每个 service 都与控制器强耦合。清单的 1.3 节说了"不再新增 physics→ui 直接调用"，但没有提到如何逐步收敛 `ctrl` 引用。建议在 80-100% 阶段增加一条：

> 逐步将 service 对 `ctrl` 的直接访问改为显式参数传入或 State 对象注入，降低 service 与控制器的耦合度。

**3. 缺少对 Mixin 架构的治理策略**

当前 UI 层大量使用 Mixin（`CircuitTabMixin`、`LoopTestTabMixin`、`WaveformTabMixin`、`WidgetBuilderMixin` 等），这些 Mixin 都通过 `self.xxx` 隐式共享宿主对象的属性。清单的 60-80% 阶段提到了"拆出 step1-5_panel"，但没有明确说 Mixin 最终要变成什么——是继续用 Mixin，还是改成组合（composition）？这个方向性决策应该在清单中记录。

**4. 回归清单（第 7 节）缺少具体预期值**

当前回归项都是定性的（"结果正确"、"正常显示"）。建议至少对核心场景补一个具体预期，例如：

> 正常场景全流程：五步流程可完成，最终成绩单 `total_score >= 80`，`veto_reason` 为空

> 事故弹窗流程：`E01` 在断路器合闸时母线无压条件下触发，弹窗出现后点击修复可消除

这样不管是人工验证还是未来写自动化测试都有据可依。

**5. 缺少 `ui/styles.py` 的拆分策略**

1012 行的样式文件是一个很容易处理但很容易被忽略的目标。建议在 80-100% 阶段加上：

> `ui/styles.py` 拆分为 `styles/theme.py`（颜色变量）、`styles/components.py`（组件级 QSS）、`styles/layout.py`（布局级样式），主文件只做 `build_app_stylesheet()` 汇总。

**6. 没有提到类型标注/静态检查**

项目已经在用 dataclass 和一些 type hints，但覆盖不完整。如果目标之一是"提高代码可靠性"，建议在 80-100% 阶段加一条：

> 核心 domain/services 层补齐类型标注，可选引入 `mypy --strict` 作为 CI 检查项。
